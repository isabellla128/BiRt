1. Abstract

ArtistSearch is a web application designed to help users explore and discover artists and their musical genres. The platform allows users to search for artists based on various filters such as genre, country, and time periods. It offers detailed artist profiles, including information about their occupations, awards, and genres. Additionally, the application provides recommendations for artists and genres, enhancing the discovery experience. Through seamless integration with external APIs and a user-friendly interface, ArtistSearch aims to provide an engaging and comprehensive tool for music enthusiasts to learn more about the global music scene.

2. Introduction

In the ever-expanding world of music, discovering new artists and genres can be a daunting task. ArtistSearch is a web-based application created to simplify this process by providing users with a comprehensive platform for exploring musical artists and their respective genres. The goal of the application is to enhance users' understanding of the global music landscape through detailed artist profiles, genre information, and recommendations based on user preferences.
The platform offers a range of filtering options, including genre, country, and time periods, enabling users to find artists that match their interests. Additionally, ArtistSearch integrates with external APIs to provide up-to-date data on artists, allowing for an immersive and data-driven exploration experience. Whether users are interested in discovering iconic artists from the past or emerging talent, ArtistSearch serves as a valuable resource for music lovers, researchers, and those eager to expand their musical horizons.
This report details the development, architecture, and functionality of the ArtistSearch application, presenting both the technical and user experience aspects that contribute to its effectiveness in making artist and genre discovery easier and more engaging.

3. Internal Data Structures/Models

This section outlines the internal data structures and models used to represent artists, genres, and their relationships within ArtistSearch.

Models

Our internal models form the foundation, allowing ArtistSearch to efficiently represent and organize data about artists, genres, and their connections.

  • Artist
    • Description: An object representing a musical artist or performer.
    • Required Fields: id, name
    • Optional Fields: photo, country, genres, birth date, death date, occupations, awards, nominations, instruments, wikipedia link
  • Genre
    • Description: An object representing a specific genre of music.
    • Required Fields: id, label
    • Optional Fields: countries, image, description, subgenres, parentCategories, superclasses, wikipediaLink
  • Artist Short Model
    • Description: An object representing a musical artist or performer.
    • Required Fields: id, name
    • Optional Fields: photo, country, genre
  • Genre Short Model
    • Description: An object representing a specific genre of music.
    • Required Fields: id, label
    • Optional Fields: countries, image
  • Artist Statistics
    • Description: A model representing statistical data about artists, grouped by country, genre, and decade.
    • Required Fields:
      • byCountry: A list of countries with the number of artists from each country.
      • byGenre: A list of music genres with the count of artists in each genre.
      • byDecade: A list of decades with the count of artists born in each decade.

APIs

The APIs in ArtistSearch enable smooth interactions, providing users with access to a vast array of artist and music genre data.

  • Artist API Endpoints
    • /artists:
      • Method: GET
      • Summary: Get a list of artists
      • Description: Retrieve a list of artists based on country, genre, and time period.
      • Parameters:
        • country: Filter artists by country.
        • genre: Filter artists by genre.
        • from-to: Filter artists by a year range (format 'YYYY-YYYY').
    • /artists/{id}:
      • Method: GET
      • Summary: Get an artist by ID
      • Description: Retrieve details of an artist based on their unique ID.
      • Parameters:
        • id: The unique identifier of the artist.
    • /artists/{id}/recommend:
      • Method: GET
      • Summary: Recommend artists based on a given artist ID
      • Description: Retrieve a list of recommended artists similar to the given artist.
      • Parameters:
        • id: The unique identifier of the artist.
    • /artists/summarize:
      • Method: GET
      • Summary: Get summarized artist information
      • Description: Retrieve a summarized view of artists based on country, genre, time period, and optional limits on the number of countries and genres.
      • Parameters:
        • country: The country to filter artists by.
        • genre: The genre to filter artists by.
        • from-to: A range of years in the format 'YYYY-YYYY'.
        • limit-countries: A limit to the number of countries to return.
        • limit-genres: A limit to the number of genres to return.
  • Genre API Endpoints
    • /genres:
      • Method: GET
      • Summary: Get a list of genres
      • Description: Retrieve a list of music genres, optionally filtered by country.
      • Parameters:
        • country: Filter genres by country.
    • /genres/{id}:
      • Method: GET
      • Summary: Get a genre by ID
      • Parameters:
        • id: The ID of the genre to fetch.
    • /genres/{id}/recommendGenres:
      • Method: GET
      • Summary: Recommend genres based on a given genre ID
      • Parameters:
        • id: The ID of the genre to fetch recommendations for.
    • /genres/{id}/recommendArtists:
      • Method: GET
      • Summary: Recommend artists based on a given genre ID
      • Parameters:
        • id: The ID of the genre to fetch artist recommendations for.

5. Linked Data Principles in ArtistSearch

ArtistSearch is built with the principles of Linked Data in mind, turning the exploration of artists, genres, and musical knowledge into a connected and discoverable journey. Here’s how the platform aligns with the core principles of Linked Data:

Use URIs as Names for Things

Each singer, and music genre on ArtistSearch is uniquely identified using URIs. These URIs ensure global recognition and persistent identification across the platform, allowing users to easily discover and explore music-related data.

Use HTTP URIs

ArtistSearch exclusively employs HTTP URIs to provide users with direct access to detailed information about artists, genres, and their relationships. These URIs are easily dereferenced, offering a seamless experience for retrieving data.

Provide Useful Information Using Standardized Formats

The platform uses RDF (Resource Description Framework) as a standardized, machine-readable format. This ensures that data about artists and genres is well-structured, easy to interpret by both humans and machines, and extensible for future enhancements.

Include Links to Other URIs

ArtistSearch integrates links within its data, forming an interconnected network of music-related information. These links allow users to explore related content, such as discovering artists from the same genre or exploring a particular artist’s career.

Link to Other Data Sources

ArtistSearch is powered by Wikidata, a trusted external data source. By linking to this vast repository of structured knowledge, the platform provides access to up-to-date and accurate data about artists, genres, and more, while supporting the open web of connected data.

Through these principles, ArtistSearch embraces the vision of Linked Data by fostering an open, connected, and accessible platform for music discovery. Users are encouraged to dive deep into a world of artists and genres where every connection tells a unique story and every link enhances the music exploration experience.

6. Web Application Development

Frameworks and Technologies Used in ArtistSearch

ArtistSearch uses a variety of powerful technologies and frameworks to deliver an optimized, scalable, and efficient music search experience. Below are some of the core technologies utilized:

Angular

Angular, a popular front-end web development framework, is used for building dynamic single-page applications (SPAs). It enables a fast and smooth user experience, particularly for handling complex views and interactions in the ArtistSearch platform. The latest version of Angular is used to ensure optimal performance and access to the most recent features and improvements.

Ionic 9 is a powerful framework used for building cross-platform mobile applications and web components. It allows for the creation of highly customizable and reusable UI components, enabling a smooth and consistent user experience across platforms.

Reasoning: Ionic was used for the standalone components in ArtistSearch due to its flexibility and ease of integration with Angular. By using Ionic 9, we were able to enhance the application's UI with responsive, mobile-first components, improving the overall user experience.

Amazon S3 for Frontend Deployment

Amazon Simple Storage Service (S3) is utilized for deploying and hosting the frontend of ArtistSearch. S3 provides scalable, durable, and low-latency storage for static website content, making it ideal for serving the frontend assets of the application.

Reasoning: S3 was chosen for its reliability, cost-effectiveness, and ease of use. By leveraging S3 for frontend deployment, we ensure fast and secure delivery of the web application, providing a seamless experience for users accessing ArtistSearch from different devices and locations.

Python

Python, a versatile and easy-to-learn programming language, is used for server-side development in ArtistSearch. It is known for its simplicity and readability, which makes it ideal for handling complex logic and backend operations.

Reasoning: Python was selected because of its rich ecosystem of libraries and frameworks that support rapid development. Its high-level nature allows for quick iterations and prototyping, essential for building a fast, scalable backend for ArtistSearch.

Flask

Flask, a lightweight and flexible Python web framework, is used to build the server-side architecture of ArtistSearch. It provides the necessary tools for creating RESTful APIs and managing client-server interactions.

Reasoning: Flask was chosen for its simplicity and extensibility, allowing us to build a robust backend API without unnecessary complexity. Its lightweight nature ensures that the server remains fast and responsive.

AWS Lambda

AWS Lambda is a serverless compute service that allows you to run code in response to specific events, such as user actions on the platform or scheduled tasks. It automatically scales according to demand without the need for server management.

Reasoning: Lambda was chosen to provide a highly scalable and cost-effective way to process background tasks and events in ArtistSearch. It ensures efficient processing while eliminating the need to manage infrastructure.

IAM Roles

Identity and Access Management (IAM) Roles are used to control access to AWS resources securely. These roles define the permissions granted to different entities, such as users or services, within the AWS infrastructure supporting ArtistSearch.

Reasoning: IAM Roles were implemented to ensure fine-grained access control within ArtistSearch's AWS environment, enhancing security by restricting access to sensitive resources based on user roles.

AWS Elastic Beanstalk

AWS Elastic Beanstalk is a platform-as-a-service (PaaS) offering that simplifies the deployment and management of applications in the cloud. It automatically handles the scaling, load balancing, and monitoring of applications.

Reasoning: Elastic Beanstalk was selected because it abstracts away infrastructure management, allowing the development team to focus on application logic while ensuring scalability and high availability for ArtistSearch.

Amazon EventBridge

Amazon EventBridge is a serverless event bus service that allows you to connect different applications using events. It is used to coordinate workflows and trigger various actions across different services within ArtistSearch.

Reasoning: EventBridge was chosen for its ability to decouple services and trigger actions based on specific events, making the system more flexible and scalable while ensuring smooth event-driven interactions within ArtistSearch.

Amazon Neptune

Amazon Neptune is a fully managed graph database service designed to work with highly connected datasets. It enables efficient querying of complex relationships between data points, such as the connections between artists, genres, and their respective attributes in ArtistSearch.

Reasoning: Neptune was chosen due to its graph-based architecture, which is ideal for representing and querying complex relationships. This allows ArtistSearch to provide fast, dynamic queries for artists, genres, and musical connections.

7. In-Depth Overview of Data Management and Processing

Collection, Transformation, and Storage of Data

ArtistSearch employs a highly efficient and secure process for collecting, transforming, and storing data. Information is retrieved from Wikidata through carefully crafted SPARQL queries executed by our Flask application hosted on AWS Elastic Beanstalk.

SPARQL, as a powerful query language, allows us to query Wikidata’s structured knowledge base efficiently. For instance, we leverage non-trivial SPARQL queries to pull data on artists, genres, and related metadata, ensuring that we gather precise and relevant information for the platform’s users.

One such complex query involves artist recommendations based on the following factors:

  • Genre Overlap: Artists who share at least one genre with the given artist are considered for recommendations.
  • Birth Year Proximity: To ensure relevance, we select artists born within a ±5 year range of the given artist.
  • Geographical Diversity: Artists from different countries are prioritized, ensuring that recommendations reflect a global perspective.

This query returns up to 10 recommended artists who meet these criteria, offering personalized and dynamic recommendations to the users.

Once the data is collected, it undergoes necessary transformations to ensure the data aligns with the frontend’s requirements before being sent to users. This transformation guarantees data consistency and optimizes it for seamless interaction on the platform.

For storage, we use AWS Neptune, a graph database that efficiently handles complex, interconnected data, ensuring reliable access and retrieval of information for the application.

Data Integrity Checks and Standardization

ArtistSearch enforces rigorous data integrity checks to ensure accuracy and reliability. Before inserting data into AWS Neptune, each data point is validated using SHACL (Shapes Constraint Language), ensuring it conforms to predefined data models and standards.

In cases where certain fields may be missing or null, default values are applied, preventing incomplete data from entering the system. This ensures that the database remains consistent and the application functions smoothly without errors.

Handling Data Errors and System Failures

ArtistSearch includes robust error handling to maintain system reliability even when unexpected issues occur. Errors encountered during data collection, transformation, or storage are captured and logged for quick troubleshooting.

When errors arise, the application returns meaningful feedback to users, ensuring they are aware of issues. Behind the scenes, the system works to resolve these errors efficiently, ensuring the platform stays stable and responsive, offering users a seamless experience.

8. Technical Details

Infrastructure Diagram

Details regarding the Backend implementation

At the backend of ArtistSearch, we’ve structured our system using microservices, which are represented by AWS Lambda functions. This architecture ensures that our application remains scalable, maintainable, and responsive, while allowing each component to perform its specific role efficiently.

Each Lambda function is designed to handle a specific operation tied to a particular route in our application. This approach divides the workload into smaller, more manageable services that can be independently scaled, modified, and updated without disrupting other parts of the system.

Here’s how our backend services interact:

Microservices and Lambda Functions:
  • We organized the backend code into microservices, each represented by a dedicated AWS Lambda function. This allows us to decouple various components of our application, providing flexibility and ease of maintenance.
  • Each Lambda corresponds to a particular route or endpoint in our application. For instance, when a user makes a request to a specific route in the frontend, the application sends the request to an EC2 instance running a Python Flask server. The Flask server then invokes the appropriate Lambda function to process the request.

Metadata Search Lambda:
  • In addition to the core Lambda functions, we have another specialized Lambda designed for searching metadata based on specific criteria, such as country or genre. This Lambda plays a vital role in gathering relevant information from the Wikidata API, as it streamlines the process of filtering the data according to user preferences and application requirements.

Lambda for Caching:
  • To optimize response times and reduce the load on external data sources, we utilize caching. We have a dedicated Lambda function that manages cache operations. When a request is made, the relevant Lambda checks if the data is available in the cache. If it is, the Lambda will retrieve the cached data to provide a fast response.
  • If the required data is not present in the cache, the Lambda triggers a process to query Wikidata for fresh data and stores the new results in both the cache and our graph database, AWS Neptune, for future use.

Scheduled Data Cleanup (EventBridge Lambda):
  • To maintain the freshness and relevance of the data in the system, we use a scheduled cleanup process. We’ve implemented an AWS EventBridge rule that triggers a Lambda function every 24 hours. This function performs a data cleanup operation on the AWS Neptune database, removing any records that are older than 24 hours. This ensures that the data remains up-to-date and relevant for the user experience, while also optimizing the performance of the database.

Request Processing Flow:
  • When a client makes a request to a route in the ArtistSearch application, the request first goes to an EC2 instance running a Python Flask server. This server acts as the middle layer, which manages incoming requests and interacts with the necessary Lambda functions.
  • The server checks whether the requested data is available in the cache by invoking the Lambda function responsible for managing cached data. If the data is found, it is returned to the user immediately, ensuring fast response times.
  • If the cache does not contain the required data, the server sends a request to Wikidata, retrieves the necessary information, and saves it both in the cache and in AWS Neptune, the graph database. This ensures that the data is readily available for future requests.

Data Management and Storage in Neptune:
  • For storing and managing data, we rely on AWS Neptune, a graph database designed to handle complex, interconnected data efficiently. It enables us to store the relationships between various data points—such as artists, albums, genres, and countries—in a way that makes it easy to query and retrieve information.
  • Data is transformed appropriately before being stored in Neptune, ensuring that it adheres to our predefined data models and is easily accessible for queries and recommendations.

CloudWatch Monitoring for System Health:
  • Since the system relies heavily on various AWS services (Lambda, EventBridge, Neptune, etc.), we utilize Amazon CloudWatch for monitoring and log aggregation. CloudWatch helps us ensure that the entire system is operating smoothly by providing real-time insights into service performance, error rates, and system health.
  • We set up monitoring across all AWS services in use. This allows us to be notified of any potential issues or failures and take corrective actions as quickly as possible. CloudWatch’s logging capabilities also enable us to trace the flow of requests and troubleshoot issues as they arise, ensuring that our backend architecture remains stable and reliable.

9. User Guide for ArtistSearch

Welcome to ArtistSearch, the ultimate platform for discovering and exploring artists, music genres, and relationships between them. This guide will help you navigate the application and utilize its features effectively.

Getting Started

Accessing the Application

  • Web Version: The frontend is accessible via a web browser.

Home Page

Upon entering the application, you will find the main navigation options that allow you to explore artists, genres, and detailed analytics about the music industry.

Main Features

Exploring Artists

The Artists Page allows users to browse a complete list of artists. Users can filter artists based on:

  • Time Period: Select artists based on their era.
  • Music Genre: Filter by the genre of music they perform.
  • Country of Citizenship: View artists based on their nationality.

Clicking on an artist opens a pop-up with detailed information about them. Additionally, users have the option to export the artist's data in RDF format using the dedicated export button.

Visualizing Artist Relationships

Users can visualize how artists relate to their music genres through an interactive graph. By clicking the designated button, a dynamic visualization is displayed, showing connections between artists and genres.

Comparing Two Artists

If two artists are selected, a comparison pop-up appears. This feature provides analytical insights into their careers, styles, and similarities, supported by graphical representations.

Statistical Analysis of Artists

On the same page, users can access summarized insights about the distribution of artists. Three graphical representations display:

  • The number of artists per country.
  • The number of artists per genre.
  • The number of artists per decade.

These visualizations dynamically update according to the applied filters.

Exploring Music Genres

The Genres Page provides a comprehensive list of music genres. A filtering option allows users to display only genres associated with a specific country. Clicking on a genre reveals more information, including all its subgenres, which can be further explored. Additionally, users have the option to export the genre's data in RDF format using the dedicated export button.

Additional Features

Intelligent Artist and Genre Recommendations

The platform suggests artists based on their similarities with a given artist, on the artist detail pop-up. The recommendation system considers:

  • Artists sharing at least one music genre with the selected artist.
  • Artists born within ±5 years of the selected artist.
  • Artists from different countries to ensure diversity.

These recommendations help users discover new artists with related styles.

Additionally, on the Genre Page, at the bottom, users can explore two recommendation lists:

  • Recommended Genres: These are selected based on genres performed by artists who also perform the currently viewed genre.
  • Recommended Artists: A list of artists who perform the selected genre, allowing users to explore musicians with similar styles.

These additional recommendations enhance the discovery experience, helping users find both new artists and related music genres effortlessly.

Tips for a Better Experience

  • Use Filters: Narrow down searches efficiently to find relevant artists and genres.
  • Explore Graphs: Utilize visualizations to understand relationships between artists and genres.
  • Check Recommendations: Discover new artists based on intelligent filtering.

Troubleshooting

For any issues, ensure you have a stable internet connection. If data appears outdated, allow some time for updates, as the system periodically refreshes its cache.
